Abstract

Ce document décrit la syntaxe basique du langage Lotus. Il décrit, en détails, la tokenisation et l'analyse syntaxique de Lotus.

Lotus est un langage open-source créée pour faciliter le travail des développeurs web et designers web. Ce document décrit sa syntaxe ainsi que la façon donc il devrait être analyser par un compilateur conforme à la spécification Lotus (donc ce module fait partie). L'interprétation de Lotus n'est, cependant, pas l'objet de ce document.

1. Introduction

ce document décrit la syntaxe du langage Lotus

Il définit aussi des algorithmes pour convertir du text (code source) en un stream de tokens, et ensuite transformer ce stream en arbre syntaxique, qui peut ensuite être analysé, vérifié, et utilisé pour différents buts par les implémenteurs, comme l'interprétation ou la traduction de code source Lotus, par exemple en HTML, CSS, et JavaScript, telle que LotusJS le fait.

Cependant, il ne décrit pas comment le stream de token ou l'arbre syntaxique peut être interprété ou analysé. Ceci peut-être décrit par d'autres documents édités par le groupe de travail lotuslang, car ce sujet n'est pas l'objet de ce document.

Étant donné que cette spécification est orienté pour les auteurs d'outils qui opèrent et intéragissent avec des documents qui utilisent la syntaxe décrite ici, celle-ci sacrifie la clarté pour la précision. Ainsi, ce document n'est pas censé être un tutoriel, et des documents plus adapté aux débutants devraient être utilisés à la place.

2. Overview

Tools implementing the Lotus syntax must use the parsing rules described in this specification to generate a parse tree that can, for example, then be used to visualize, analyze, optimize or translate Lotus source code.

2.1 Definitions

This section defines terms used throughout this document.

code point
A Unicode code point. A Unicode character with a value between 0 (hexadecimal) and FFFF (hexadecimal).
input stream
A stream of code points.
consume the next input code point
Pop a code point from the input stream and return it. The next input code point is now consumed and has the status of current input code point, and the value of the next input code point needs to be updated accordingly.
next input code point
The first code point in the input stream that has not been consumed yet.
current input code point
The last code point to have been consumed.
reconsume the current code point
Push the current input code point on top of the input stream, so that the next time that you are instructed to consume the next input code point, the current input code point will be consumed instead.
EOF code point
A conceptual code point used to represent the end of the input stream.
digit
A code point between U+0030 DIGIT ZERO (0) and U+0039 DIGIT NINE (9).
hex digit
A digit, or a code point between U+0041 LATIN CAPITAL LETTER A (A) and U+0046 LATIN CAPITAL LETTER F (F), or a code point between U+0061 LATIN SMALL LETTER A (a) and U+0066 LATIN SMALL LETTER F (f).
number
A number then has a numeric part, comprised either of one or more digits optionally followed by U+002E FULL STOP (.) and one or more digits; or of U+002E FULL STOP (.) and one or more digits.
A number can optionally end with a "scientific notation" part, comprised of U+0045 LATIN CAPITAL LETTER E (E) or U+0065 LATIN SMALL LETTER E (e), optionally followed by U+002B PLUS SIGN (+) or U+002D HYPHEN MINUS (-), followed by one or more digits.
uppercase letter
A code point between U+0041 LATIN CAPITAL LETTER A (A) and U+005A LATIN CAPITAL LETTER Z (Z).
lowercase letter
A code point between U+0061 LATIN SMALL LETTER A (a) and U+007A LATIN SMALL LETTER Z (z).
letter
An uppercase or lowercase letter.
non-ASCII code point
A code point with a value equal or greater than U+0080 CONTROL.
name code point
A letter, a digit, a non-ASCII code point, or a U+005F LOW LINE (_).
newline
A code point with a value of U+000A LINE FEED (represented as the escape sequence \n in most programming languages).
whitespace
A newline, a U+0009 CHARACTER TABULATION, or a U+0020 SPACE.
maximum allowed code point
The code point with the value U+FFFF.
input token stream
The stream of token produced by the tokenizer.
current input token
The last token from the input token stream to have been consumed.
next input token
The first token in the input token stream that has not been consumed yet.
EOF token
A conceptual token used to represent the end of the input token stream.
reconsume the current input token
Push the current input token on top of the input token stream, so that next time you are instructed to consume the next input token, the current input token will be consumed instead.

2.2 Token types

Implementations should use the algorithms described in Section 4 Tokenization, or act as if they use them (i.e. the resulting token stream should contain at least the same information, no matter the underlying implementation). The tokenization phase should output a stream of zero or more of the following tokens.

<delim-token>
A token holding a single code point as its value. It can represent any code point.
<whitespace-token>
A token representing one or more whitespace.
<logical-or-token>
A token representing a logical OR operator (U+007C VERTICAL LINE U+007C VERTICAL LINE (||)).
<logical-and-token>
A token representing a logical AND operator (U+0026 AMPERSAND U+0026 AMPERSAND (&&)).
<logical-xor-token>
A token representing a logical XOR operator (U+005E CIRCUMFLEX ACCENT U+005E CIRCUMFLEX ACCENT (^^)).
<equality-token>
A token representing an equality operator (U+003D EQUALS SIGN U+003D EQUALS SIGN (==)).
<less-equal-token>
A token representing a less-or-equal operator (U+003C LESS-THAN SIGN U+003D EQUALS SIGN (<=)).
<greater-equal-token>
A token representing a greater-or-equal operator (U+003E GREATER-THAN SIGN U+003D EQUALS SIGN (>=)).
<not-equal-token>
A token representing a not-equal operator (U+0021 EXCLAMATION MARK U+003D EQUALS SIGN (!=)).
<decrement-token>
A token representing a decrement operator (U+002D HYPHEN MINUS U+002D HYPHEN MINUS (--)).
<increment-token>
A token representing an increment operator (U+002B PLUS SIGN U+002B PLUS SIGN (++)).
<arrow-token>
A token representing an arrow operator (U+003D EQUALS SIGN U+002B PLUS SIGN (=>)).
<number-token>
A token representing a number. It holds one value: the numeric value stored as a sequence of code points.
<string-token>
A token representing a string. It holds two value: the content of the string as a sequence of code points, and the code point delimiting the string.
<ident-token>
A token representing an identifier. It holds one value : the name of the identifier, stored as a sequence of code points.

3. Formal Syntax Definition

This section formally describes Lotus' syntax through EBNF and railroad diagrams. An informal and more human-readable definition of the syntax can be found here.

3.1 EBNF Conventions

This specification uses a certain set of the EBNF syntax and extends it. Following is a definition of the expressions used in this section and Section 3.2 Syntax Definition.

#uN (where N is an hexadecimal integer)
matches the code point with a value of N.
[#uN1-#uN2] where N1 and N2 are two hexadecimal integer.
matches any code point with a value comprised in the range between N1 and N2 (inclusive).
[#uN1-#uN2#uN3-#uN4] where N1, N2, N3 and N4 are four hexadecimal integer.
matches any code point with a value comprised in the ranges between N1 and N2 (inclusive) or N3 and N4 (inclusive).
[c1-c2] where c1 and c2 are two code points
matches any code point beetween c1 and c2 (inclusive).
[c1-c2c3-c4] where c1, c2, c3 and c4 are four code points
matches any code point between c1 and c2 (inclusive) or between c3 and c4 (inclusive).
expression
can be a non-terminal symbol, a list of non-terminal symbol, a list of expressions, or any of the expressions defined in this section. It matches any text conforming to the symbol(s) or expression(s).
^(expression)
matches any expression that doesn't match expression.
[expression]
matches expression or nothing, i.e. expression is optional.
{expression}
matches zero or more occurences of expression.
<token(expression)>
matches a token with the name token with a value that matches expression
"string"
matches any sequence of code points matching the string given inside the double quotes.
'string'
matches any sequence of code points matching the string given inside the single quotes.
expressionA expressionB
matches expressionA followed by expressionB.
expressionA | expressionB
matches expressionA or expressionB
N * expression (where N is an integer)
matches N repetition of expression

NOTE

/* ... */ is used for comments.

EXAMPLE

Take the following grammar : 

code_point = [#u0030-#uFFFF];
string = "'" {code_point} "'";

digit = [0-9];
number = ["+" | "-"] (digit {digit} ["." digit {digit}] | "." digit {digit}) [("e" | "E") ["+" | "-"] digit {digit}];

letter = [a-zA-Z];
identifier = (letter | "_") {letter | digit | "_"};

value = identifier | number | string;

This grammar defines a format for strings, for numbers, etc... Below is a table showcasing some example of sequence that would match a production rule of that grammar. An explanation for some cases are written under the table.

Rule name Example 1 Example 2 Example 3 Invalid case
code_point h + Ø 𐀟 (U+1001F)1
string 'hello' '' (empty string)2 '16€ is equal to $18,08 as of 2/07/19' 'world3
digit 0 8 5 f4
number 15.6e-3 -63 +.75E-85 -.e+5.06
letter A k M ø7
identifier point_1 _m_arrayStart SavedFile 6th_person8
value 60 'some text' hello +9

Note 1: The code point 𐀟 has a value of U+1001F. However, code_point is defined as any code point between U+0030 and U+FFFF, and U+1001F is greater than U+FFFF.

Note 2: Even though there isn't any code_point between the two delimiters ('), it is still valid, since the grammar allows empty string : it matches zero or more occurence of code_point between the delimiters ('). Therefore, '' is a valid string.

Note 3: This case is invalid since the string rule requires that it must end with the delimiter ('), which is not the case here.

Note 4: f is not in the range [0-9], since this range translate to [#u0030-#u0039], which matches any code point with a value in the range U+0030 and U+0039, and f has a value of U+0066.

Note 5: This one may seem weird, but it is a valid number (0.0000000075 in fact). So, lets take our rule : 

number = ["+" | "-"] (digit {digit} ["." digit {digit}] | "." digit {digit}) [("e" | "E") ["+" | "-"] digit {digit}]

The first part of this rule (["+" | "-"]) means that a number can optionally start with either "+" or "-". In our case, our number starts with a "+". Next, we have : 

(digit {digit} ["." digit {digit}] | "." digit {digit})

This means that a number must have either one or more digits, optionally followed by "." and one or more digit; or "." followed by one or more digits. In our case, our number match the second possibility ("." followed by digits), and so it is valid. We then have : 

[("e" | "E") ["+" | "-"] digit {digit}]

This part is optional, as indicated by the square brackets ([...]). It indicates that a number can end with "e" or "E", optionally followed by "+" or "-", followed by one or more digits. Here we have "E" followed by - and the number "8"

We just disected our number : "+" - ".75" - "E-8" !

Note 6: By applying the same steps as above, we can see that we have a problem with the second and third parts. The second part doesn't have a digit after the ".", and the third part has a decimal part (the ".0" at the end), which doesn't match our rule.

Note 7: The code point U+00F8 (ø) is outside of the range [a-zA-Z].

Note 8: In this case, the input starts with a digit, which doesn't match the rule for identifier, since it says it needs to start wiht either a letter or "_".

Note 9: + is not a value, because it is neither an identifier, number, nor a string.

4. Tokenization

Tokenization is a process by which an algorithm converts a sequence of code points into a list tokens.

A token is an object representing zero or more code points and optionally metadata associated with it.

EXAMPLE

A <number-token> represents a number.

It holds a sequence of code points. This sequence is the token's "metadata", or "value"

The sequence of code points is its representation.

The following algorithms describes how to transform a stream of code points, designated as the input stream, into a stream of tokens.

Implementations should act as if they used those algorithms, i.e. the resulting token stream should contain at least the same information as if it implemented the following algorithms. See the Lotus tokenizer reference implementation for more information, and see the test suite to test the conformance of your implementation.

4.1 Preprocessing the input stream

Before feeding the input stream into the tokenizer, it needs to be processed as following :

  • Replace any U+0000 NULL code point with U+FFFD REPLACEMENT CHARACTER

  • Replace any U+000D CARRIAGE RETURN code point (\r), U+000C FORM FEED code point (\f), or pair of U+000D CARRIAGE RETURN U+000A LINE FEED code points (\r\n) by a single U+000A LINE FEED code point (\n).

4.2 Consume a token

This section describes how to consume a token from a stream of code points.

This algorithm returns a single code point.

Consume the next input code point from the input stream.

whitespace
Consume as much whitespace as possible and return a <whitespace-token>.
U+0022 QUOTATION MARK (") or U+0027 APOSTROPHE (') or U+0060 GRAVE ACCENT (`)
Consume a string token with the ending delimiter set to the current input code point. Return the resulting <string-token>.
digit
Reconsume the current code point. Consume a number token and return the resulting <number-token>.
U+002E FULL STOP (.)
Check if the input stream would start a number, If yes, reconsume the current code point, consume a number token and return the resulting <number-token>. Otherwise, return a <delim-token> with its value set to the current input code point.
U+002D HYPHEN MINUS (-)

If the next input code point is U+002D HYPHEN MINUS (-), consume it and return a <decrement-token>

Otherwise, return a <delim-token> with its value set to the current input code point.
U+002B PLUS SIGN (-)

If the next input code point is U+002E PLUS SIGN (+), consume it and return a <increment-token>

Otherwise, return a <delim-token> with its value set to the current input code point.
U+005F LOW LINE (_)
Reconsume the current code point. Consume an identifier token and return the resulting <ident-token>.
letter
Reconsume the current code point. Consume an identifier token and return the resulting <ident-token>.
U+0023 NUMBER SIGN (#)

If the two next input code points are U+0023 NUMBER SIGN U+0023 NUMBER SIGN (##)

  1. Consume them.
  2. While the three next input code points are not U+0023 NUMBER SIGN U+0023 NUMBER SIGN U+0023 NUMBER SIGN (###), consume the next input code point.
  3. Consume the three next input code point (which should be "###").
  4. Return a <whitespace-token>.

Otherwise,

  1. While the next input code point is not a newline, consume it.
  2. Consume the next input code point.
  3. Return a <whitespace-token>.

NOTE

This specification allows implementations to choose to preserve comments and still be compliant. To do so, implementations can introduce a <comment-token> that holds two value : the sequence of code points that constitues the comment (content), and the delimiter (delimiter) as a sequence of one or more code points.

Then, the algorithm becomes :

Let representation be an empty string.

If the two next input code points are U+0023 NUMBER SIGN U+0023 NUMBER SIGN (##)

  1. Consume them.
  2. While the three next input code points are not U+0023 NUMBER SIGN U+0023 NUMBER SIGN U+0023 NUMBER SIGN (###), consume the next input code point and append it to representation.
  3. Consume the three next input code point (which should be "###").
  4. Return a <comment-token> with content set to representation and delimiter set to "###".

Otherwise,

  1. While the next input code point is not a newline, consume it and append it to representation.
  2. Consume the next input code point.
  3. Return a <comment-token> with content set to representation and delimiter set to "#".

U+007C VERTICAL LINE (|)
If the next input code point is U+007C VERTICAL LINE (|), consume it and return a <logical-or-token>.
U+0026 AMPERSAND (&)
If the next input code point is U+0026 AMPERSAND (&), consume it and return a <logical-and-token>.
U+005E CIRCUMFLEX ACCENT (^)
If the next input code point is U+005E CIRCUMFLEX ACCENT (^), consume it and return a <logical-xor-token>.
U+003D EQUALS SIGN (=)

If the next input code point is U+003D EQUALS SIGN (=), consume it and return a <equality-token>.

Otherwise, if the next input code point is U+003E GREATER-THAN SIGN (>), consume it and return a <arrow-token>.

U+003C LESS-THAN SIGN (<)
If the next input code point is U+003D EQUALS SIGN (=), consume it and return a <less-equal-token>.
U+003E GREATER-THAN SIGN (>)
If the next input code point is U+003D EQUALS SIGN (=), consume it and return a <greater-equal-token>.
U+0021 EXCLAMATION MARK (!)
If the next input code point is U+003D EQUALS SIGN (=), consume it and return a <not-equal-token>.
EOF
Return an EOF token.
anything else
Return a <delim-token> with its value set to the current input code point.

4.3 Consume a string token

This section describes how to consume a string token from a stream of code points.

This algorithms should be called with an ending delimiter, which denotes the code point that ends the string. If no ending delimiter is specified, use the current input code point instead.

This algorithm returns a single <string-token>.

Let representation be an empty string.

Repeatedly consume the next code point from the input stream.

EOF
Throw an "Unexpected end of file" exception.
ending delimiter
Return a <string-token> and let its value be representation and its ending delimiter be ending delimiter.
anything else
Append the current input code point to representation.

4.4 Consume a number token

This section describes how to consume a number token from a stream of code points.

This algorithm returns a single <number-token>.

NOTE

This algorithm assumes that the input stream can start a valid number.

  1. Let representation be an empty string.

  2. While the next input code point is a digit, consume it and append it to representation.

  3. If the two next input code point are a single U+002E FULL STOP (.) followed by a digit.
    1. Consume the two next code points and append them to reprensentation.

    2. While the next input code point is a digit, consume it and append it to representation.

      NOTE

      If the U+002E FULL STOP was not followed by a digit throw an "Invalid number formatting" exception.

  4. If the next input code point is U+0045 LATIN CAPITAL LETTER E (E) or U+0065 LATIN SMALL LETTER E (e), consume it and append it to representation, then:
    1. Check if the three next input code point would start a number.

    2. If the check failed, then throw an "Invalid number formatting" exception and exit.

    3. Otherwise (i.e. if the check succeeded), if the next input code point is U+002B PLUS SIGN (+) or U+002D HYPHEN MINUS (-), consume it and append it to representation.

    4. While the next input code point is a digit, consume it and append it to the representation.

Return a <number-token> and let its value be representation.

4.5 Consume an identifier token

This section describes how to consume an indentifier token from a stream of code points.

This algorithm returns a single <ident-token>.

NOTE

This algorithm assumes that the input stream can start a valid identifier.

Let representation be an empty string.

While the next input code point is a valid name code point, consume it and append it to representation.

Return an <ident-token> and let its value be representation.

4.6 Check if a code point would start an identifier

This section desbribes how to check if a code point would start an identifier. This algorithm can be called with a code point, or from the input stream.

NOTE

This algorithm doesn't consume any code point. If it is called from the input stream, it should push back onto the top of the stream any code point consumed.

Look at the code point.

U+005F LOW LINE (_)
Return true.
letter
Return true.
digit
Return false.
EOF
Return false.
anything else
Return false

4.7 Check if two code points would start a number

This section describes how to check if three code points would start a number. This algorithm can be called with three code points, or from the input stream.

NOTE

This algorithm doesn't consume any code point. If it is called from the input stream, it should push back onto the top of the stream any code point consumed.

Look at the first code point.

U+002E FULL STOP (.)
If the second code point is a digit, return true. Otherwise, return false.
digit
Return true.
anything else
Return false.
